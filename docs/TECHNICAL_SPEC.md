---

## 1. Назначение системы

Интеллектуальный мобильный ассистент предназначен для помощи людям с нарушениями зрения в ориентировании в пространстве и выполнении бытовых задач. Система анализирует видеопоток с камеры смартфона и сообщает пользователю о значимых объектах, препятствиях и текстовой информации посредством голосовых подсказок через наушники.

---

## 2. Целевая аудитория

- Люди с полной или частичной потерей зрения
- Люди с прогрессирующими заболеваниями зрения
- Пожилые люди с ухудшением зрительных функций

По данным ВОЗ, сотни миллионов людей в мире имеют нарушения зрения различной степени тяжести. Разработка подобного ассистента способна существенно повысить уровень самостоятельности пользователей в повседневной жизни.

## 3. Функциональные требования

### 3.1 Модуль детекции объектов

**Цель:** Обнаружение и локализация объектов в кадре в режиме реального времени.

**Требования:**
- Детекция объектов из стандартных классов (COCO): люди, транспорт, мебель, бытовые предметы, препятствия
- Скорость обработки: ≥ 15 FPS на мобильном устройстве
- Порог уверенности (confidence threshold): ≥ 0.5
- Подтверждение объекта через N последовательных кадров (стабилизация)
- Алгоритм Non-Maximum Suppression (NMS) для фильтрации дублирующих рамок
- Letterbox-препроцессинг для сохранения пропорций изображения

**Входные данные:** Кадр с камеры (RGB, переменное разрешение)  
**Выходные данные:** Список объектов — класс, координаты bounding box, уверенность

---

### 3.2 Модуль оценки глубины (Depth Estimation)

**Цель:** Определение расстояния до обнаруженных объектов для формирования пространственных подсказок.

**Требования:**
- Монокулярная оценка глубины (без стереокамеры или LiDAR)
- Поддерживаемые модели: MiDaS, ZoeDepth, DPT
- Формирование пространственных описаний: «Препятствие впереди через ~2 метра», «Лестница справа»
- Предупреждения об опасных объектах с указанием направления и дистанции

**Входные данные:** Кадр + bounding boxes из модуля детекции  
**Выходные данные:** Словарь {объект → расстояние, направление}

---

### 3.3 Модуль OCR (распознавание текста)

**Цель:** Распознавание текста на вывесках, ценниках, упаковках товаров, документах.

**Требования:**
- Pipeline: обнаружение области текста → выравнивание перспективы → распознавание символов → постпроцессинг
- Постпроцессинг: нормализация регистра, удаление лишних пробелов, исправление опечаток, сверка со словарём
- Работа с кириллицей и латиницей
- Устойчивость к плохому освещению, наклону и частичному перекрытию

**Входные данные:** Кадр или вырезанная область  
**Выходные данные:** Распознанная строка текста

---

### 3.4 Модуль распознавания денежных купюр

**Цель:** Быстрое и надёжное определение номинала банкноты российских рублей.

**Требования:**
- Поддерживаемые номиналы: 50, 100, 200, 500, 1000, 2000, 5000 ₽
- Гибридный алгоритм:
  1. Детекция банкноты в кадре (YOLO)
  2. OCR для считывания номинала с купюры
  3. CNN-классификатор как подтверждающий этап
- Озвучивание результата: «Пять тысяч рублей»
- Порог уверенности для озвучивания: ≥ 0.85 (высокое требование к надёжности)

**Входные данные:** Кадр с банкнотой  
**Выходные данные:** Номинал (строка) + уровень уверенности

---

### 3.5 Модуль распознавания лекарств

**Цель:** Идентификация лекарственных препаратов по упаковке с нечётким поиском.

**Требования:**
- OCR названия препарата с упаковки
- Нечёткий поиск (fuzzy matching) по локальной базе лекарственных средств
- Алгоритм: расстояние Левенштейна / token sort ratio (RapidFuzz)
- Возврат топ-3 совпадений с процентом сходства
- База данных: ГРЛС или эквивалентный источник (SQLite, офлайн)
- Устойчивость к частично нечитаемым или повреждённым упаковкам

**Входные данные:** Текст с OCR (название препарата)  
**Выходные данные:** Список совпадений — название, производитель, % сходства

---

### 3.6 Модуль приоритизации событий

**Цель:** Предотвращение перегрузки пользователя информацией — озвучивать только важные события.

**Требования:**
- Иерархия приоритетов (убывающий порядок):
  1. Опасные препятствия (ступени, открытые люки, транспорт)
  2. Люди (особенно в непосредственной близости)
  3. Запрошенные пользователем объекты (купюры, лекарства, текст)
  4. Фоновые объекты (мебель, предметы интерьера)
- Антиспам: один и тот же объект не озвучивается чаще одного раза в N секунд
- Формирование связных фраз вместо перечисления: «Перед вами дверь на расстоянии двух метров» вместо «Дверь. Стена. Стул.»

---

### 3.7 Модуль синтеза речи (TTS)

**Цель:** Озвучивание результатов анализа через наушники.

**Требования:**
- Задержка от события до начала озвучивания: < 500 мс
- Основной режим: офлайн TTS (pyttsx3)
- Дополнительный режим: edge-tts при наличии интернета (более естественный голос)
- Язык: русский (основной), английский (дополнительный)
- Регулируемая скорость речи

## 4. Нефункциональные требования

### 4.1 Производительность
- FPS основного pipeline: **≥ 15 FPS** (комфортная работа), **≥ 30 FPS** (целевое значение)
- Потребление RAM: **< 1.5 ГБ** в активном режиме
- Потребление CPU/GPU: оптимизировано под мобильные SoC (Snapdragon, Dimensity)

### 4.2 Автономность
- Все вычисления выполняются локально на устройстве
- Интернет не требуется для основных функций
- Офлайн-база лекарств и купюр

### 4.3 Надёжность
- Подтверждение объектов через несколько кадров перед озвучиванием
- Использование порогов уверенности для каждого модуля
- Явное уведомление пользователя при низкой уверенности распознавания

### 4.4 Устойчивость к условиям съёмки
- Плохое освещение (ночь, тень)
- Motion blur при движении
- Шум изображения
- Частичное перекрытие объектов

## 6. Архитектура YOLO (детали)

Используется семейство моделей YOLO (You Only Look Once), выполняющих детекцию за один проход сети:

- **Backbone** — извлечение иерархических признаков из изображения (текстуры → формы → объекты)
- **Neck (FPN / PAN)** — объединение карт признаков разных масштабов для детекции объектов разного размера
- **Head** — предсказание координат bounding box, уверенности и класса объекта

Многоуровневая структура важна для одновременного обнаружения как мелких объектов (текст на вывеске, светофор вдали), так и крупных (автомобиль, человек рядом).

---

## 7. OCR Pipeline (детали)

1. Обнаружение текстовой области (text detection)
2. Выравнивание перспективных искажений (deskewing)
3. CNN или Transformer-энкодер → последовательность признаков
4. Декодер (CTC / attention) → символьная последовательность
5. Постпроцессинг: исправление опечаток, нормализация, сверка со словарём

## 9. Известные технические сложности

- **Плохое освещение и motion blur** — ухудшают качество OCR и детекции
- **Производительность мобильных устройств** — ограниченные RAM, CPU, батарея
- **Надёжность OCR** — ошибки в сложных шрифтах, ракурсах, отражениях
- **Реальное время** — все модули должны работать в едином pipeline без блокировок
- **Безопасность** — ошибки распознавания препятствий могут быть опасны для пользователя
